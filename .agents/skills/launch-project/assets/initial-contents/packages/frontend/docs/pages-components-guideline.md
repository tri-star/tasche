# ページ、コンポーネント実装ガイドライン

## ページレベルのコンポーネント作成に関するガイドライン

### 全般

- React Router ではページを Route Component として扱う
- 1 URL に対して 1 Route Component を基本とし、URL ごとの責務は routes 配下の Route Component に置く
- routes 配下は薄く保ち、表示と画面構成に集中させる
  - 役割は画面構成、ルート単位のデータ取得、遷移制御に限定する
- 複雑な表示ロジックは feature 配下のコンポーネントへ切り出す
- loading, error, empty state を考慮する
- ルート固有の権限チェックやパラメータ解釈はこの層で扱う

## パーツレベルのコンポーネント作成に関するガイドライン

## 全般

- 1 コンポーネント 1 責務を基本とする
- props で入出力が分かる設計にする
- 再利用前提の UI はドメイン依存を持たせすぎない
- 条件分岐が増えたら分割を検討する
- データ取得と描画責務を必要以上に混ぜない

### アクセシビリティ

- ボタン、リンクとして表現する要素にはbutton, aを使用する
- ラジオボタン、チェックボックス、ドロップダウンを装飾する際は、タブやキーボード操作が可能になるように意識する
- ダイアログにはdialogタグを利用する
- モバイル端末での操作性を確保するため、インタラクティブな要素は上下左右に広めの余白を確保する

### コンポーネントのユニットテスト

- それぞれのコンポーネント(`component.tsx`)は同階層に(`__tests__/component.test.tsx`) を作成し、ユニットテストを記述する
  - テストコードをコロケーションする
- API呼び出しを伴うコンポーネントはmswを利用してレスポンスをモックしてテストする
  - レスポンスの型がany型になるとレスポンス型の不一致に気付かずメンテが難しくなるため、orvalの生成する型を利用してレスポンスの型を指定しながらテストコードを書く

### ベースコンポーネント

ボタン、チェックボックスなど基礎となるコンポーネントでは次の点に注意する。

- 原則として、要素の幅、マージンは利用する側のコンポーネントが指定する思想で設計する
- tailwind-merge, tailwind-variantsを利用してコンポーネントのバリアントを表現する

#### コンポーネントのバリアント

ベースとなるUIのバリアントはデザイントークンの定義に基づく。各コンポーネントは、対応するセマンティックトークンが定義されているバリアントのみを実装する。

- バリアント（配色）はコンポーネントごとにデザイントークンで定義される
  - 例: Button は primary, secondary, error, ghost
  - 例: Tag は default, primary, success, error, warn, info, purple, pink, orange
  - 例: Card は default, selected

- 各バリアントの状態もデザイントークンで定義される
  - default: 通常状態
  - hover: ホバー状態
  - active: アクティブ状態
  - disabled: 無効状態

- バリアントの定義は `tailwind-variants` の `tv()` で宣言的に管理し、`cn()` で外部classNameと結合する

### パーツコンポーネント

ベースコンポーネントだけでページを実装すると1つのページが肥大化し易く、共通化も難しいため、
ページ内にセクションが複数ある場合や繰り返し登場するUI(カード、テーブルの行)は
1つにまとめてパーツコンポーネントを作成する。

パーツコンポーネントは、ページレベルのコンポーネントと同じ階層に `components` フォルダを作成し、その中に配置するか、共通化する場合は `features/components` 配下に配置する。
