# Human Review Required

## 目的

コード上の単純な path 判定だけでは捕捉しきれない「必ず人間レビューが必要な条件」を定義する。

この文書の条件に合致すると AI が判断した場合、最終 verdict は `human-review` とする。

## 条件

### 1. 認証・認可・権限境界の仕様判断が含まれる

次のような変更は、実装が一見正しく見えても仕様判断を含むため人間レビューを必須とする。

- ログインフローの変更
- 権限チェック条件の変更
- ユーザーに見えるデータ範囲の変更
- 管理者権限の付与や制限条件の変更

### 2. データ互換性や移行方針の判断が含まれる

次のような変更は、コードだけでは安全性を断定しにくいため人間レビューを必須とする。

- schema 変更
- migration の追加や修正
- 既存データの削除や変換
- 一意制約、index、外部キーの変更

### 3. 性能要件の判断が必要な変更

次のような変更は、実データ量や運用条件を踏まえた判断が必要なため人間レビューを必須とする。

- 大量データに対するクエリ戦略の変更
- 集計処理や一覧取得ロジックの変更
- キャッシュ有無で挙動が大きく変わる変更

### 4. 公開 API や外部契約の変更

次のような変更は、利用者影響の確認が必要なため人間レビューを必須とする。

- API request/response 仕様の変更
- route path や parameter の変更
- 外部連携インターフェースの変更

### 5. 重大インシデントに繋がり得る運用変更

次のような変更は、コード上は問題がなくても運用判断を要するため人間レビューを必須とする。

- CI/CD の publish, deploy 条件変更
- 秘密情報や認証情報の扱い変更
- feature flag や環境切替条件の変更

## 補足

- この文書に書かれた条件は、コード上の bug が見つかった時だけでなく、仕様判断や運用判断が必要な時にも適用される
- AI がコードだけでは妥当性を断定できない場合は、この文書を根拠に `human-review` としてよい
